# Design — v0

Procedural Line Rider track generation synchronized to music. This is the canonical design for v0. See `DECISIONS.md` for the reasoning behind each choice and the alternatives considered.

## Architecture

Two-layer pipeline:

```
Spec  ──►  compiler  ──►  Track + DriftReport
```

An optional `adjuster` layer above the compiler (deferred to v0.1+) would explore spec variations using the compiler as a black box. Out of scope for v0.

The compiler keeps `lr-core` in the loop throughout. Every geometric decision is validated against the real physics engine before commit. There is no approximate-physics planning anywhere in the pipeline — the engine is the oracle.

## Spec

A TypeScript module exporting a `Spec` object. Authored by hand for v0; eventually generated by a music-analysis script.

### Schema

```ts
type Spec = {
  duration: number;              // seconds
  defaults?: Partial<SectionAxes>;
  contacts: Contact[];           // hard sync events
  sections: Section[];           // soft style blocks
};

type Contact = {
  t: number;                     // seconds, must be in [0, duration]
};

type Section = {
  t0: number;                    // seconds
  t1: number;                    // seconds
} & SectionAxes;

type SectionAxes = {
  air?:           number;        // [0, 0.99]
  speed?:         number;        // [0, 1]
  contact_style?: number;        // [0, 1]
  grain?:         number;        // [0, 1]
};
```

### Hard constraints

Three constraints the compiler must satisfy, or fail audibly. Each violation appears in the DriftReport as a distinct hard failure, not aggregated with soft drift.

1. **Contact precision.** Each `Contact(t)` must coincide with a rider ground-contact event within ±1 frame.
2. **Survival.** The rider must survive — not be ejected, not flythrough, not terminate prematurely — through every section.
3. **No off-beat landings.** Every landing event in the produced track must coincide with a `Contact` (within ±1 frame). The rider may slide on a catch line for some frames after the beat-aligned landing (that's covered by `contact_style`), but the *landing event itself* — the transition from airborne to grounded — is forbidden at any non-Contact frame. Un-musical landings ruin sync; this constraint guards against them.

### Soft targets

Per-section axis values are targets the compiler aims for. Achieved values appear in the DriftReport. The user reads the report and iterates the spec. No automatic tolerance gating in v0.

### Per-axis precise definitions

Each axis is a float with a precise, measurable definition on a finished track.

`air ∈ [0, 0.99]`
- Definition: fraction of frames in the measured range where the rider is airborne (no sled point in contact with any line).
- Clamped to ≤ 0.99 so `contact_style` remains defined: a fully airborne section has no contacts to measure.

`speed ∈ [0, 1]`
- Definition: `mean(|rider_velocity|) / SPEED_CAP`, averaged over frames in the measured range.
- `1.0` ≈ near-terminal rider. `0.0` ≈ barely-moving rider.
- `SPEED_CAP` is a calibrated constant (TODO).

`contact_style ∈ [0, 1]`
- Definition: per ground-contact event, measure `traversed_length / segment_total_length` over the contact's duration; average across the measured range's contacts.
- `0` = pure deflection (rider uses ≤ 10% of any contacted segment).
- `1` = full-segment rides.
- Undefined for ranges with zero contacts; the compiler skips this axis when undefined.

`grain ∈ [0, 1]`
- Definition: `median(line_length) / LINE_LENGTH_CAP` over the measured range's lines.
- `0` = very short segments (~5 units). `1` = long segments (~50 units).
- `LINE_LENGTH_CAP` is a calibrated constant (TODO).

The four axes are not strictly orthogonal. `air` and `speed` are physically coupled — high airborne fraction requires launches, which require speed, and airborne velocity is governed by ballistics. `grain` is a property the compiler controls directly through line geometry, while `air`, `speed`, and `contact_style` are rider-state measurements the compiler influences indirectly. When axis targets conflict at the extremes, the DriftReport surfaces achieved values honestly and the user iterates. The vocabulary is committed for v0 on the understanding that a system robust to non-orthogonal axes is also confidence-building toward later refinement.

### Layering semantics

Sections may overlap arbitrarily — nested, partial, or both. The effective axis value at any time `t` is computed by walking all sections containing `t` in spec-declaration order; **last-defined wins per axis**. Axes not specified by any covering section fall back to `defaults`, then to compiler-internal defaults.

### Variety

The compiler carries a seeded RNG threaded throughout. Identical Spec + identical seed → byte-identical Track. Identical Spec + different seed → genuinely different but spec-conforming Track.

Variety is produced at two levels:

1. **Cross-gap target sampling.** Within a section, individual gaps (intervals between consecutive hard Contacts) sample their own per-axis target from a Gaussian centered on the section's target with spread σ. Consecutive gaps feel different even when the section axis value is constant.
2. **Within-gap candidate diversity.** For each gap, the compiler generates many candidate Arc placements via wide random parameter sampling; among those satisfying the hard constraints, the lowest-cost one wins. Because the candidate distribution is wide, surviving candidates can be qualitatively different shapes.

For v0, σ is a hidden compiler constant (default suggested: 0.15 for [0,1] axes). In v0.1, σ becomes the user-facing `regularity` axis.

## Compiler

### Atomic placement primitive: Arc

```ts
type Arc = {
  anchor: { x: number; y: number };
  length: number;                  // total arc length, units
  startAngleDeg: number;           // tangent at start
  endAngleDeg: number;             // tangent at end (== startAngle ⇒ straight)
  segments: number;                // polyline granularity
  curveBias: number;               // [-1, +1], shape of curvature profile
};
```

`curveBias = 0` produces a circular interpolation between start and end tangents. `curveBias > 0` biases the bend toward the start (eased exit, S-curve shapes). `curveBias < 0` biases toward the end (eased entry, sharper exit). Range and exact semantics: TODO calibrate.

An Arc compiles mechanically to `segments` Line primitives. The compiler reasons about Arcs; the engine consumes Lines.

### Terminology

- **Spec** — user-authored intent
- **Section** — a spec block (t0, t1, axis values)
- **Contact** — a hard sync event in the spec
- **Gap** — compiler-internal interval between consecutive Contacts (including head and tail)
- **Arc** — compiler's placement primitive (parametric)
- **Line** — engine-level geometric output

### Compiler I/O contract

**Input**:
- `Spec`
- `seed: number` (optional, default 0)
- engine instance

**Output**:
- `Track` — `{ lines: Line[] }`
- `DriftReport`

DriftReport schema:

```ts
type DriftReport = {
  contacts: {
    t_target: number;
    t_actual: number | null;       // null if missing
    frame_error: number | null;
    status: "hit" | "drift" | "missing";
  }[];
  sections: {
    section_index: number;
    survived: boolean;
    axes: {
      [axis: string]: { target: number; achieved: number; error: number };
    };
  }[];
  off_beat_landings: { frame: number }[];   // any landing event not aligned with a Contact
  terminus: { frame: number; reason: string };
};
```

### Initial conditions

For v0: rider starts at origin `{ x: 0, y: 0 }` with engine-standard initial velocity (~0.4 px/frame). Single start state per spec; no per-section override.

### Per-gap budget compilation

The core algorithm.

**1. Slice the timeline.** Hard Contacts divide the spec's `[0, duration]` into Gaps: a head gap from `t=0` to the first Contact, interior gaps between consecutive Contacts, and a tail gap from the last Contact to `duration`.

**2. Compute per-gap targets.** For each Gap and each axis specified by covering Section(s):
- Section target for the axis = `T`.
- Gap target = sample from `Normal(T, σ²)`, clipped to the axis range.
- This produces cross-gap variation even when the section target is constant.

**3. Compile each gap.** For each Gap in time order:
- **Inputs.** Known start rider state (from previous gap's end, or initial conditions for the head gap), known end-frame Contact constraint (or end-of-spec for the tail), per-gap axis targets.
- **Generate K candidates.** Each candidate is a sequence of one or more Arcs that bridge the gap. Parameters are drawn from wide uniform distributions over Arc parameter bounds — anchor offset relative to the rider's predicted trajectory, length, start/end angles, segment count, curveBias. The bounds are tied loosely to the rider's current state to keep candidates plausibly relevant, but the sampling is otherwise unbiased — no archetype palette, no axis-conditioned distribution.
- **Contact precision via bisection.** For the Arc nearest the gap's end-Contact, bisect an Arc parameter (typically anchor Y) so the engine confirms the landing within ±1 frame. Bisection assumes monotonicity locally; when the search diverges or oscillates (a non-monotone parameter-vs-frame region), fall back to a coarse grid search over the parameter range. If neither converges, the candidate fails the hard gate.
- **Hard gate.** Simulate each candidate forward through the engine. Reject candidates that lose the rider (terminus), produce a landing event at any non-Contact frame, or fail to hit the gap's end-Contact within ±1 frame.
- **Rank survivors.** Compute per-candidate cost = sum across axes of `|achieved_axis − gap_target_axis|`. Pick the lowest-cost survivor; commit.
- **Backtrack on failure.** If no candidate survives after K tries, retry the previous gap with its second-best candidate (so this gap inherits a different rider end-state). Up to 2 levels of cross-gap backtrack. If failure persists, record a hard failure for this gap in the DriftReport and continue with the best-surviving state from the most recent attempt.

**4. Assemble.** Concatenate all Arcs, generate Line geometry, return Track + populated DriftReport.

Gaps are not subdivided by synthetic anchors. If a gap is too long for the rider to traverse without an off-beat landing — e.g., a 10-second airborne stretch that physics won't permit without crashing — the candidate generator will fail to produce hard-gate survivors and the DriftReport will surface the infeasibility. The user iterates the spec, typically by adding intermediate Contacts (which then are beats with musical justification) or by relaxing axis targets.

### Budget

For v0: wall-clock timeout = `2 × duration(spec)`. Compiler aborts and reports partial results if exceeded.

## Calibration constants

These need empirical tuning against rendered tracks. Marked clearly in code as `TODO calibrate`:

- `SPEED_CAP` — px/frame, divisor in `speed` axis
- `LINE_LENGTH_CAP` — units, divisor in `grain` axis
- `σ` — default cross-gap target sampling spread
- `K` — per-gap candidate budget
- Arc parameter bounds — anchor offset range, length range, start/end angle ranges, segment count range
- Arc `curveBias` interpretation and range

## First milestone

A 10-second spec with:
- 5 Contacts at `t ∈ {2.0, 4.0, 6.0, 8.0, 9.5}`
- One Section over `[0, 10]` with `air = 0.7`

Produces a renderable Track and a populated DriftReport. Small enough to debug end-to-end; big enough to validate the architecture.
