# Design — current state (in-progress)

## Architecture

```
Spec  ──►  compiler  ──►  Track + DriftReport
```

Two layers. Optional `adjuster` layer above the compiler (deferred to v0.1+) would explore spec variations using the compiler as a black box.

The compiler has `lr-core` in the loop throughout — every geometric decision is validated against the real physics engine before commit. No approximate-physics planning. The engine is the oracle.

## Spec (authoring surface)

A TypeScript module exporting a `Spec` object. Authored by a user; eventually generated by a music-analysis script.

```ts
type Spec = {
  duration: number;              // seconds
  defaults?: Partial<Section>;   // baseline axis values
  contacts: { t: number }[];     // hard sync events
  sections: Section[];           // soft style blocks, may stack per-axis
};

type Section = {
  t0: number; t1: number;        // seconds
  air?:           number;        // [0, 1]
  altitude?:      number;        // [-1, +1]
  contact_style?: number;        // [0, 1]
  grain?:         number;        // [0, 1]
};
```

### Hard vs soft

- **Contacts (hard).** Rider must be in ground contact at `t` within ±1 frame. Violations appear in DriftReport as sync failures.
- **Section axes (soft).** Compiler targets; achieved values reported per section. No automatic tolerance gating in v0 — user reads the report and iterates the spec.

### Per-axis precise definitions

- `air ∈ [0, 1]`: fraction of the section's frames where the rider is airborne (no sled point in contact with any line). 0 = always in contact, 1 = always airborne.
- `altitude ∈ [-1, +1]`: `(y_end − y_start) / (g · duration² / 2)`. -1 = drop equivalent to free fall over the section's duration. +1 = symmetric rise of the same magnitude. 0 = no net altitude change.
- `contact_style ∈ [0, 1]`: per contact event, measure `traversed_length / segment_total_length`; average across the section's contacts. 0 = pure deflection (rider uses ≤10% of any contacted segment). 1 = full-segment rides. Undefined for sections with zero contacts.
- `grain ∈ [0, 1]`: `median(line_length) / line_length_cap`. 0 = very short segments (~5 units). 1 = long segments (~50 units). `line_length_cap` is a fixed empirical constant.

### Layering

Sections may stack in time. Per-axis: a section overrides the default for axes it specifies; unset axes inherit. Last-defined wins for direct conflicts. So a baseline `{air: high}` over the whole timeline plus a `{altitude: drop}` override on a sub-range gives "drop within an aerial baseline" without either layer needing to know the other exists.

### Variety

- The compiler carries a seeded RNG threaded throughout.
- Same Spec + same seed → byte-identical Track (reproducible).
- Same Spec + different seed → genuinely different but spec-conforming Track.
- A future `regularity` axis (v0.1+) would control *how much* spread the RNG-driven choices have. Deferred.

## Compiler

### Atomic placement primitive: Arc

```ts
type Arc = {
  anchor: { x: number; y: number };
  length: number;
  startAngleDeg: number;
  endAngleDeg: number;     // == startAngleDeg ⇒ straight line
  segments: number;        // polyline granularity
};
```

An Arc compiles mechanically to N `Line` geometric segments. The compiler reasons about Arcs; the engine consumes Lines.

Terminology, distinct names at each level:
- **Spec** — user-authored intent
- **Section** — a spec block (t0, t1, axis values)
- **Contact** — a hard sync event in the spec
- **Arc** — compiler's placement primitive (parametric)
- **Line** — engine-level geometric output

### Compiler I/O (locked)

- **Input**: Spec + optional seed (default 0) + engine instance.
- **Output**: Track (`{lines: Line[]}`) + DriftReport.

DriftReport structure:
- Per Contact: `{t_target, t_actual, frame_error, status: "hit" | "drift" | "missing"}`.
- Per Section: per-axis `{target, achieved, error}` and `survived: bool`.
- Terminus: `{frame, reason}` if the rider died early.

### Compiler internals — UNRESOLVED

Two approaches were explored in the design conversation; both were rejected:

1. **Hill-climbing**. Generate a small set of candidate Arcs per placement, score against per-section axis distance, pick the best. Failure mode: converges to a single locally-optimal Arc shape and places it repeatedly. Produces visually uniform tracks. Earlier attempts to fix this via fitness terms (the old coolScore) did not work and added their own complexity.

2. **Sampling from distributions centered on axis targets**. Draw Arc parameters from per-axis distributions, validate by simulation. Failure mode: only works for `grain` (where the Arc length IS the axis being measured). For `air`, `altitude`, and `contact_style`, the measurement is a section-wide statistic that depends on multiple Arcs interacting; no single Arc parameter directly drives the measurement. Sampling can produce variety but cannot reliably achieve a section's target value.

The right approach is unresolved. The compiler must:
- Enforce hard Contact constraints (likely via bisection on Arc parameters — a proven technique).
- Achieve soft axis targets at section granularity (real optimization or search needed).
- Produce visually varied output (no convergence to uniform repeated shapes).
- Keep the rider alive across sections.
- Use the engine in the loop throughout.

### Budget

For v0: `timeout = 2 × duration(spec)` (e.g., 60 seconds compute for a 30-second spec). Generous; tightened when data shows what's needed.

## First milestone (proposed)

10-second spec with 5 Contacts plus one Section with `air = 0.7`. Produces a renderable Track plus DriftReport with per-Contact frame errors and per-Section axis errors. Small enough to debug; big enough to validate the architecture end-to-end.

## Repo shape (proposed)

New `scripts/v0/` directory alongside existing code. Existing bench / strategy / primitive code stays untouched until v0 produces tracks we like, then is deprecated in a sweep.
